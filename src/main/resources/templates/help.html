Help & Strategy Guide
Trading strategy, setup rules, quantity management & page documentation
Trading Strategy Overview

TraderEdge is a CPR (Central Pivot Range) breakout trading bot for NSE F&O stocks. It scans for narrow and inside CPR stocks from the daily bhavcopy, monitors real-time price action on configurable timeframe candles, and enters trades when CPR levels are broken with confirming conditions.

How It Works

  • Daily Setup: Downloads NSE bhavcopy, computes CPR levels (Pivot, TC, BC, R1-R4, S1-S4, PDH, PDL) for all F&O stocks
  • Filtering: Selects stocks with narrow CPR (high breakout potential) and inside CPR (price compressed within prior day's range)
  • Z-Score Classification: Each stock's daily range is classified using 20-day history: NS Narrow + Small Range (z < -1.5) | NL Narrow + Large Range | IS Inside + Small Range | IL Inside + Large Range
  • Breakout Detection: On each candle close, checks if price broke above/below CPR levels with green/red candle confirmation
  • Entry: Market order placed, SL + Target(s) placed on fill
  • Exit: SL hit, target hit, trailing SL, or auto squareoff at configured time

Two Trading Modes

ModeDescription
Trend FollowingTrade in the direction of the breakout. Standard CPR level breakouts.
Mean ReversionTrade back toward value. Magnet trades (IV/OV days) and EV reversal trades.

Signal Source

All signals are generated by the bot's internal breakout scanner (CandleAggregator + BreakoutScanner). Real-time WebSocket ticks are aggregated into configurable timeframe candles, and breakouts are detected on each candle close.

Open Classification (IV / OV / EV)

The first candle close of the day determines the open classification, which drives how the bot trades for the rest of the session.

Breakout Rules

  • Green candle required for buys (close > open), red candle for sells
  • Priority: highest level wins (R4 > R3 > R2 > R1/PDH > CPR)
  • Each level fires only once per day per symbol — cleared when position closes to allow re-entry
  • Day High/Low breakouts: when price breaks the session high/low without a CPR level break, fires BUY_ABOVE_DH or SELL_BELOW_DL. Lowest priority — CPR breakouts take precedence. Only after Opening Range locks.
  • VWAP/ATP check (if enabled): buy requires close > ATP, sell requires close < ATP

Target Calculation Flow

Targets go through a two-step process before the trade is accepted:

StepWhat HappensExample (BUY)
1. CPR TargetDefault target set from CPR level (R1, R2, etc.)Entry 1850, target R2 = 1875
2. Day High/Low ShiftIf session high is between entry and target, shift target to day high (always on, no toggle)Day high 1865 is between 1850-1875 → target shifts to 1865
3. Small Target FilterIf final target < N x ATR from entry → skip trade (configurable, can disable)ATR=20, threshold=1.0. Distance 1865-1850=15 < 20 → SKIP

This ensures every trade has enough profit room to justify the risk. Day high/low acts as an intraday resistance — even if the CPR target is further away, the day high/low is the realistic barrier price needs to overcome first.

Magnet Trades (Mean Reversion)

Magnet trades are mean-reversion setups where price is pulled back toward the CPR (the "magnet"). Active on IV and OV days.

SetupTriggerTargetLogic
BUY_ABOVE_S1_PDLClose above S1 and PDL (from below)CPR (min TC/BC)Price pulled back from below value, bouncing toward CPR
SELL_BELOW_R1_PDHClose below R1 and PDH (from above)CPR (max TC/BC)Price pulled back from above value, falling toward CPR

Quantity Rules

  • IV magnet — full qty (stock within value, high conviction)
  • OV magnet — half qty (stock has momentum, magnet may fail)

Guard

Close must be between S1/R1 and CPR — the target (CPR) must still be ahead, not already passed.

On Extended Value (EV) days, when the Opening Range (OR) breaks counter to the gap direction, the bot trades mean-reversion back toward value with closer targets.

Quantity Stacking Order

Each step multiplies the running qty from the previous step. Minimum qty = 2 (rounded to even).

#StepLogic
1Base qtyrisk per trade / SL distance. Or fixed qty if configured.
2Capital capmin(base qty, (capital per trade x leverage) / price)
3Extreme level÷2 if breakout at R2+ / S2+ level
4Session move %÷2 if price moved > N% from day open / PDC
5ProbabilityHPT = 1x, MPT = configurable, LPT = configurable
6Level factorR3/S3 = configurable (0.75x), R4/S4 = configurable (0.5x)
7Mean-reversion÷2 for all EV reversals + OV magnet trades

Pre-Qty Filters (reject before qty calculation)

  • EV day + inside OR range — trade skipped entirely
  • EV day + OR still forming — trade skipped
  • Large candle filter — reject if candle move > N x ATR from level
  • Small candle filter — reject if candle move < N x ATR
  • Volume filter — reject if candle volume < N x average volume
  • Small target filter — reject if final target < N x ATR from entry (after day high/low shift)
  • R4/S4 disabled — reject R4/S4 signals
  • ATR invalid — reject if ATR is zero or NaN
Split Targets (T1 + T2)

When enabled, every position gets two target orders: T1 at a partial distance and T2 at the full target. Each gets half the quantity.

How It Works

  • T1 price = entry + (target distance x T1%) — e.g., 50% of the way to target
  • T2 price = full target (100%)
  • T1 qty = total / 2 (rounded to even)
  • T2 qty = total - T1 qty

Fill Scenarios

EventActionTrade Record
T1 hitsCancel SL, place new SL at entry (breakeven) for remaining qtyPartial: T1 qty, reason=TARGET_1
T2 hitsCancel SL, position fully closedPartial: T2 qty, reason=TARGET_2
SL before T1Cancel both T1 and T2Full qty, reason=SL
SL after T1 (breakeven)Cancel T2Remaining qty, reason=SL_BREAKEVEN

Settings

  • Enable Split Targets — toggle on/off
  • T1 Distance — 25% / 50% / 75% of target distance
  • Min Distance — only split when target distance > N x ATR (0 = always split)
  • Requires qty ≥ 4 for split (otherwise single target)

After T1 hit, trailing SL continues from breakeven and keeps tightening.

Stop Loss & Trailing SL

Initial SL

SL = Entry ± ATR(period) x Multiplier. Recalculated from actual fill price (not signal close price).

Trailing SL (ATR Peak-Based)

  • Tracks peak price (LONG) or trough price (SHORT) since entry
  • New SL = peak - ATR x trailing multiplier (LONG) or trough + ATR x trailing multiplier (SHORT)
  • Activation threshold: trailing only starts after price moves N x ATR in profit (default 1.0 ATR)
  • SL only tightens — never loosens (LONG: SL moves up, SHORT: SL moves down)
  • Modified on Fyers via order modification API (not cancel/replace)

Auto Square Off

All open positions squared off at configured time (e.g., 15:10). SL and target orders cancelled, market exit orders placed.

First Candle Close09:20 / 09:30Depends on scanner timeframe (5 min = 09:20, 15 min = 09:30). Open classification (IV/OV/EV) is determined. First candle close persisted for OR and gap detection.

Timeline

09:15 OPEN → 09:20-09:30 1st CANDLE (IV/OV/EV set) → 09:20 TRADING START → 09:45 OR LOCKS → 09:20-14:45 ACTIVE TRADING → 14:45 TRADING END → 15:10 AUTO SQUAREOFF → 15:30 MARKET CLOSE

Key Timing Rules

  • Before trading start: signals are rejected ("outside trading hours")
  • Before first candle close: open classification is unknown, OR is "FORMING". Non-EV trades proceed normally. EV detection not yet possible.
  • Before OR locks (EV day): EV signals are skipped ("OR still forming"). Non-EV signals are unaffected.
  • After trading end: no new entries. Trailing SL continues. SL/target orders remain active until hit or squareoff.
  • Auto squareoff: runs once at configured time. Cancels all pending SL/target orders, then places market exit orders for each open position.

Configuring Times

All times are configurable in Settings > Hours tab. Set trading start after your scanner timeframe's first candle (e.g., 15-min candles = start at 09:30+ to ensure first candle has closed). Set auto squareoff before 15:30 to avoid exchange-level forced closure.

Order Management & Trade Lifecycle

1. Signal to Order

When the breakout scanner detects a valid signal, it passes through SignalProcessor for validation and quantity calculation. If approved:

  • A market order (MPP — Market Protection Price) is placed via Fyers API
  • Order type: 2 (Market) | Side: 1 (BUY) or -1 (SELL) | Product: INTRADAY
  • The entry order ID is registered for fill monitoring

2. Entry Fill Detection

Two parallel paths monitor the entry order:

PathHowSpeedWhen Active
WebSocket (OrderEventService)Fyers Order WebSocket pushes order status changes in real-timeInstant (~100ms)When Order WS is connected
Polling (PollingService)Polls Fyers order book every 2 seconds checking order status2-10 secondsFallback when WS is down

On fill detection (status = 2 / Traded):

  • Actual fill price (tradedPrice) extracted from order book — never uses Fyers blended netAvgPrice
  • SL recalculated from actual fill price: SL = fillPrice ± ATR × multiplier
  • Position state saved to database (symbol, side, qty, entry, SL, target)
  • PositionManager updated (LONG/SHORT/NONE tracker)
  • Market data subscriptions updated to include new symbol

3. SL + Target Placement (OCO)

Immediately after entry fill, SL and target orders are placed:

OrderTypeQtyPrice
Stop LossType 3 (Stop-Market)Full qtyEntry ± ATR × multiplier (rounded to tick)
Target (single)Type 1 (Limit)Full qtyComputed target price (rounded to tick)
T1 (split)Type 1 (Limit)Half qtyEntry + N% of target distance
T2 (split)Type 1 (Limit)Half qtyFull target price
  • Retry logic: up to 3 attempts. If one order succeeds and the other fails, the successful one is cancelled and both retried together.
  • Fallback: if all retries fail, scanForManualOCO() scans the order book for manually placed SL/target orders matching the symbol.
  • Order IDs saved to position state for tracking and restart recovery.

4. OCO Monitoring (One-Cancels-Other)

Once SL + Target are placed, they are monitored as an OCO pair — when one fills, the other is cancelled.

PathHowWhat It Detects
WebSocketOrder status push events from FyersFill (status 2), cancellation (status 1), rejection (status 5), modification (price change on status 6)
PollingPolls order book every 5 secondsSame statuses, plus detects manual price modifications

5. Fyers Order Status Codes

CodeMeaningBot Action
1CancelledOrder was cancelled (by bot, user, or exchange). If SL cancelled without fill, position may be unprotected.
2Traded (Filled)Order executed. For entry: place SL+Target. For SL/Target: cancel counterpart, record trade.
5RejectedOrder rejected by exchange. Logged as error. Entry rejection stops the trade. SL rejection = unprotected position.
6Pending (Open)Order is live, waiting to fill. Bot monitors for status change. Detects price modifications.

6. OCO Fill Handling

Single Target Mode:

EventAction
SL fillsCancel target order. Record trade (reason: SL or TRAILING_SL). Clear position state.
Target fillsCancel SL order. Record trade (reason: TARGET). Clear position state.

Split Target Mode (T1 + T2):

EventAction
T1 fillsCancel current SL (full qty). Place new SL at breakeven (entry price) for remaining qty. Record partial trade (TARGET_1). Update position state.
T2 fillsCancel SL. Record partial trade (TARGET_2). Clear position state.
SL fills (before T1)Cancel both T1 and T2. Record full trade (SL). Clear position state.
SL fills (after T1)Cancel T2. Record remaining qty trade (SL_BREAKEVEN). Clear position state.

7. Trailing SL Modification

  • MarketDataService monitors LTP on every WebSocket tick
  • Tracks peak price (LONG) or trough price (SHORT) since entry
  • When profit exceeds activation threshold (N × ATR), trailing begins
  • New SL = peak - ATR × trailing multiplier (only tightens, never loosens)
  • Fyers API: PUT /api/v3/orders/sync modifies the SL order price in-place (no cancel/replace)
  • Updated SL price saved to position state and tracked in OCO context

8. Manual Modifications

If you manually modify SL or target price on Fyers terminal:

  • Both WebSocket and polling paths detect price changes on pending orders (status 6)
  • New prices are synced back to position state (saveOcoState())
  • OCO context updated so cancellation uses correct counterpart
  • Trailing SL respects manual SL — it only tightens from the current price, never loosens it

9. Position Sync (Safety Net)

Every 10 seconds, syncPosition() compares broker positions with bot state:

  • Detects new positions opened outside the bot (manual trades)
  • Detects positions closed externally (manual squareoff)
  • Resolves entry price: in-memory → disk → order book lookup → Fyers avg (last resort)
  • Auto-scans for SL/target orders on newly detected positions
  • Skipped entirely when both WebSockets are connected (zero API polling)

10. Square Off Process

Triggered manually (button) or automatically at configured time:

  • For each open position: cancel all pending SL/target orders first
  • Place market exit order (opposite side of position)
  • Wait for exit fill, record trade with reason "AUTO_SQUAREOFF" or "MANUAL"
  • Clear all position state
  • Telegram notification sent

11. Crash Recovery

On restart, the bot recovers state from the database:

  • Position state (symbol, side, qty, entry, SL, target, order IDs) loaded from SQLite
  • OR state (opening range, first candle close, day open) loaded from JSON file
  • Scanner state (broken levels, signal history) loaded from JSON file
  • Active positions re-detected via syncPosition()
  • OCO monitoring resumed for positions with saved SL/target order IDs
  • Split target state (T1 filled, remaining qty) preserved

Order Flow Diagram

SIGNAL → SignalProcessor (validate, qty, target) → MARKET ORDER placed → WAIT FOR FILL (WS push or 2s poll) → FILLED (extract tradedPrice) → Recalculate SL from fill price → PLACE SL + TARGET (retry x3) → MONITOR OCO (WS push or 5s poll) → SL/TARGET FILLS → Cancel counterpart → Record trade → Clear state

Trend, Alignment & Probability

Two Timeframes

The bot uses two timeframes to determine trade quality:

TimeframeSourceWhat It Tells You
Weekly Trend (Larger)LTP vs weekly CPR levels (weekly OHLC aggregated from daily candles)The bigger picture — is the stock in a multi-day uptrend, downtrend, or range?
Daily Trend (Smaller)LTP vs today's daily CPR levels (TC, BC, Pivot)Today's intraday bias — is price above or below today's value area?

How Weekly Trend Is Calculated

TrendConditionMeaning
Strong BullishLTP > weekly R1 AND LTP > previous week highStock has broken above weekly resistance — very strong uptrend
BullishLTP > weekly CPR top (TC)Stock above weekly value area — uptrend
NeutralLTP within weekly CPR rangeNo clear directional bias
BearishLTP < weekly CPR bottom (BC)Stock below weekly value area — downtrend
Strong BearishLTP < weekly S1 AND LTP < previous week lowStock has broken below weekly support — very strong downtrend

How Daily Trend Is Calculated

TrendConditionMeaning
Strong BullishLTP > R1 AND LTP > PDHPrice well above today's value — strong intraday momentum up
BullishLTP > daily CPR top (TC)Price above today's value area
NeutralLTP within daily CPR rangePrice inside today's value — no intraday bias
BearishLTP < daily CPR bottom (BC)Price below today's value area
Strong BearishLTP < S1 AND LTP < PDLPrice well below today's value — strong intraday momentum down

Trend Alignment

When the weekly trend and the breakout direction agree, the trade has higher conviction. When they disagree, conviction is lower. The daily trend is informational — it shows where price is right now relative to today's CPR, but does NOT affect probability classification.

Weekly TrendDaily TrendBreakoutAlignmentInterpretation
BullishBullishBuyFull alignmentStrongest setup — both timeframes + breakout agree. HPT.
BullishBearishBuyPartialWeekly says up but today is weak. Could be a dip-buy opportunity. Still HPT (weekly drives probability).
BearishBearishBuyFull misalignmentBoth timeframes bearish but buying. Counter-trend. LPT.
NeutralAnyEitherNeutralWeekly has no bias. Breakout could go either way. MPT.

Probability = Weekly Trend vs Breakout Direction

The probability classification is only based on weekly trend vs breakout direction. Daily trend is NOT used. Both "Bullish" and "Strong Bullish" count as bullish for probability (same for bearish).

Weekly TrendBreakout DirectionProbabilityWhy
Strong Bullish / BullishBuyHPTBuying with the larger trend — highest conviction
Strong Bearish / BearishSellHPTSelling with the larger trend — highest conviction
NeutralEitherMPTNo weekly bias — could go either way
Strong Bearish / BearishBuyLPTBuying against the weekly downtrend — counter-trend
Strong Bullish / BullishSellLPTSelling against the weekly uptrend — counter-trend

Why Weekly Trend Matters More Than Daily

  • Weekly trend reflects the structural bias over 5 trading days — institutional flows, sector rotation, multi-day momentum
  • Daily trend is just where price is right now vs today's pivot — it can flip multiple times in a day
  • A stock can be daily bearish (price below today's CPR) but weekly bullish (above weekly CPR) — this means today is a pullback in an uptrend, which is actually a buying opportunity
  • The reverse is also true: daily bullish but weekly bearish = a bounce in a downtrend, selling opportunity

Risk Management

Risk Budget

The bot enforces a daily risk budget to prevent catastrophic losses.

ParameterFormulaExample
Max Daily LossTotal Capital x Max Risk Per Day %₹5,00,000 x 2% = ₹10,000
Exposed RiskSum of (entry - SL) x qty for all open positions3 positions = ₹2,000 + ₹1,500 + ₹1,800 = ₹5,300
Consumed RiskRealized losses from closed trades today2 losing trades = ₹1,200 + ₹800 = ₹2,000
Total RiskExposed + Consumed₹5,300 + ₹2,000 = ₹7,300
Remaining BudgetMax Daily Loss - Total Risk₹10,000 - ₹7,300 = ₹2,700 remaining

When total risk reaches the daily limit, new signals are rejected ("risk exposure limit reached"). Existing positions continue to be managed.

Risk Per Trade

Each trade's risk is capped at a fixed rupee amount:

  • Base qty = Risk Per Trade / SL Distance
  • Example: Risk ₹500, SL distance ₹20 → qty = 25
  • This ensures no single trade can lose more than the configured amount (before slippage)

Capital Per Trade

A secondary cap on position size:

  • Capital cap qty = (Capital Per Trade x Leverage) / LTP
  • Final qty = min(risk qty, capital cap qty)
  • Prevents outsized positions on low-price stocks where risk-based qty might be very large

Exposed Risk Updates

Exposed risk is recalculated in real-time as SL prices change:

  • Trailing SL tightens → exposed risk decreases
  • SL moves above entry (breakeven) → exposed risk = 0 for that position
  • T1 hit → SL at breakeven, remaining position is risk-free
  • Displayed on Home dashboard and Positions page hero cards

Risk Layers

LayerWhat It Protects Against
SL per tradeSingle trade max loss (ATR-based)

Capital per tradePrevents outsized positions on cheap stocks

Safety Guards

  • Kill switch — admin toggle on Positions page. When OFF, no new trades accepted. Existing positions still managed.
  • Dedup guard — 5-second window prevents duplicate trade recordings from race conditions
  • Min qty = 2 — positions always have at least 2 shares (allows even split for T1/T2)
  • Tick rounding — all prices rounded to exchange tick size (from Fyers symbol master)
  • Crash recovery — position state persisted to SQLite, OCO order IDs survive restarts
